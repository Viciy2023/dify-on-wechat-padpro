# OpenClaw + WeChatPadPro 整合方案（方案 A）

> 基于苍何教程 + 源码分析，将 WeChatPadPro iPad 协议直接集成到 OpenClaw 插件中，
> 替换原有的付费代理 API，实现免费微信机器人。

## 一、项目现状分析

### 1.1 openclaw-wechat（苍何版）

苍何的教程描述了他的实现思路：
- 使用 TypeScript 编写，与 OpenClaw 原生技术栈一致
- 三层架构：消息接收层 → 中转网关层 → OpenClaw 对接层
- 已实现：消息去重（LRU）、@触发+私聊直连、"思考中..."状态提示、多模型路由

**核心文件结构：**
```
openclaw-wechat-main/
├── index.ts                    # 插件入口，注册 channel
├── openclaw.plugin.json        # 插件元数据
├── package.json                # 依赖和 openclaw 插件声明
└── src/
    ├── channel.ts              # Channel 注册 + 账号解析
    ├── bot.ts                  # 消息处理核心（去重、路由、分发）
    ├── proxy-client.ts         # ★ 代理 API 客户端（需要替换的部分）
    ├── callback-server.ts      # Webhook 接收服务器
    ├── reply-dispatcher.ts     # 回复分发器
    ├── config-schema.ts        # 配置 schema
    ├── runtime.ts              # 运行时环境
    └── types.ts                # 类型定义
```

**关键问题：** `proxy-client.ts` 调用的是苍何的付费代理服务（需要 apiKey），
所有 API 都经过 `proxyUrl` 中转，接口格式为：
- `POST /v1/iPadLogin` → 获取二维码
- `POST /v1/getIPadLoginInfo` → 检查登录
- `POST /v1/sendText` → 发送文本
- `POST /v1/sendImage2` → 发送图片
- `POST /v1/webhook/register` → 注册 webhook

### 1.2 dify-on-wechat-padpro（WeChatPadPro 协议）

这是一个 Python 项目，直接对接 WeChatPadPro 8059 协议：

**核心文件结构：**
```
dify-on-wechat-padpro-main/
├── app.py                              # 主入口
├── config.py                           # 配置管理
├── channel/wechatpadpro/
│   ├── wechatpadpro_channel.py         # ★ 通道实现（4800+ 行）
│   └── wechatpadpro_message.py         # 消息模型
└── lib/wechatpadpro/WechatAPI/
    └── Client8059/
        ├── __init__.py                 # WechatAPIClient8059 组合类
        ├── base.py                     # API 基类（HTTP 调用封装）
        ├── login.py                    # 登录：二维码、二次登录、唤醒登录
        ├── message.py                  # 消息：文本、图片、语音、视频、表情
        ├── friend.py                   # 好友管理
        ├── chatroom.py                 # 群聊管理
        ├── tool.py                     # 工具：下载图片/语音/媒体
        ├── tool_extension.py           # WebSocket 消息监听
        └── ...
```

**8059 协议 API 格式（直接 HTTP 调用，无需代理）：**
- `GET  /login/GetLoginQrCode?key=xxx` → 获取二维码
- `POST /login/LoginCheckQR?key=xxx` → 检查登录
- `POST /login/LoginTwiceAutoAuth?key=xxx` → 二次登录
- `POST /message/SendMessage?key=xxx` → 发送消息
- `GET  /message/GetSyncMsg?key=xxx` → 拉取消息（HTTP 轮询）
- `WS   /ws/GetSyncMsg?key=xxx` → 实时消息（WebSocket）

**认证方式：** URL 参数 `?key=TOKEN_KEY`，TOKEN_KEY 由 ADMIN_KEY 生成

## 三、需要修改的文件清单与改动分析

### 3.1 ★ 核心替换：proxy-client.ts → padpro-client.ts

这是最关键的改动。原来的 ProxyClient 调用苍何的付费代理，
新的 PadProClient 直接调用本地 8059 协议服务。

**接口映射表：**

| 功能 | 原 ProxyClient (付费) | 新 PadProClient (免费 8059) |
|------|----------------------|---------------------------|
| 认证方式 | Header X-API-Key | URL 参数 ?key=TOKEN_KEY |
| 获取二维码 | POST /v1/iPadLogin | GET /login/GetLoginQrCode?key=xxx |
| 检查登录 | POST /v1/getIPadLoginInfo | POST /login/LoginCheckQR?key=xxx |
| 二次登录 | 无 | POST /login/LoginTwiceAutoAuth?key=xxx |
| 发送文本 | POST /v1/sendText | POST /message/SendMessage?key=xxx |
| 发送图片 | POST /v1/sendImage2 | POST /message/SendMessage?key=xxx (MsgType=2) |
| 获取联系人 | POST /v1/getAddressList | GET /friend/GetFriendList?key=xxx |
| 注册 Webhook | POST /v1/webhook/register | 不需要（改用 WebSocket/轮询） |
| 账号状态 | POST /v1/account/status | GET /user/GetUserInfo?key=xxx |
| 生成 Key | 无（代理分配） | POST /GenAuthKey?key=ADMIN_KEY |

**请求体格式差异（以发送文本为例）：**

```
原 ProxyClient:
  POST /v1/sendText  body: { wcId, content }

新 PadProClient（8059 协议格式）:
  POST /message/SendMessage?key=xxx
  body: { MsgItem: [{ ToUserName: wcId, MsgType: 1, TextContent: content, AtWxIDList: [] }] }
```

### 3.2 ★ 核心改造：callback-server.ts → message-listener.ts

**原方案：** 苍何的代理服务通过 Webhook 主动推送消息到插件的 HTTP 服务器。

**新方案：** 8059 协议不支持 Webhook 推送，需要改为：
- 方式一（推荐）：WebSocket 监听 — 连接 ws://127.0.0.1:8059/ws/GetSyncMsg?key=xxx
- 方式二：HTTP 轮询 — 定时调用 GET /message/GetSyncMsg?key=xxx

**消息格式转换：**

8059 协议返回的消息：
```json
{
  "MsgId": "xxx",
  "FromUserName": "wxid_xxx",
  "ToUserName": "wxid_yyy",
  "MsgType": 1,
  "Content": "消息内容",
  "CreateTime": 1234567890
}
```

需要转换为 WechatMessageContext（bot.ts 期望的格式）：
```json
{
  "id": "xxx",
  "type": "text",
  "sender": { "id": "wxid_xxx", "name": "wxid_xxx" },
  "recipient": { "id": "wxid_yyy" },
  "content": "消息内容",
  "timestamp": 1234567890000,
  "threadId": "wxid_xxx"
}
```

**群消息识别：** 8059 协议中，群消息的 FromUserName 是群 ID（xxx@chatroom），
Content 格式为 "发送者wxid:\n消息内容"，需要解析拆分。

### 3.3 小改：channel.ts

- 配置字段从 apiKey + proxyUrl 改为 adminKey + host + port
- resolveWeChatAccount 中移除 proxyUrl 必填校验
- 新增 adminKey 必填校验

### 3.4 小改：reply-dispatcher.ts

- 将调用 ProxyClient.sendText() 改为调用 PadProClient.sendText()
- 接口签名基本一致，改动很小

### 3.5 小改：types.ts

- ResolvedWeChatAccount 中 proxyUrl → host + port
- ProxyClientConfig → PadProClientConfig
- 新增 adminKey 字段

### 3.6 不变的文件

- index.ts — 插件入口，无需改动
- bot.ts — 消息处理核心（去重、路由、分发），完全不变
- runtime.ts — 运行时环境，不变

## 四、关键技术难点

### 4.1 消息接收方式的根本变化

这是最大的架构差异。原方案是被动接收（Webhook），新方案需要主动拉取（WebSocket）。

影响范围：
- callback-server.ts 需要重写为 WebSocket 客户端
- 需要处理 WebSocket 断线重连
- 需要处理消息格式转换（8059 → WechatMessageContext）

参考 dify-on-wechat-padpro 的实现：
- wechatpadpro_channel.py 的 _message_listener() 方法
- tool_extension.py 的 connect_websocket() 方法

### 4.2 登录流程差异

原方案只有扫码登录，8059 协议支持三种登录方式：
1. 扫码登录 — 首次使用
2. 二次登录 (LoginTwiceAutoAuth) — 已登录过，自动恢复
3. 唤醒登录 (AwakenLogin) — 设备唤醒

需要实现完整的登录状态管理，参考 wechatpadpro_channel.py 的 _auto_login() 方法。

### 4.3 TOKEN_KEY 管理

8059 协议需要 ADMIN_KEY 来生成 TOKEN_KEY：
- 首次使用：用 ADMIN_KEY 调用 GenAuthKey 生成 TOKEN_KEY
- TOKEN_KEY 有有效期，需要定期续期
- 需要持久化保存 TOKEN_KEY，避免每次重启都重新生成

### 4.4 群消息解析

8059 协议的群消息格式特殊：
```
FromUserName: "xxx@chatroom"
Content: "wxid_sender:\n实际消息内容"
```
需要解析出真实的发送者和消息内容。

## 五、配置方案

```jsonc
// ~/.openclaw/openclaw.json
{
  "channels": {
    "wechat": {
      "enabled": true,
      "host": "127.0.0.1",       // 8059 协议服务地址
      "port": 8059,               // 8059 协议服务端口
      "adminKey": "YOUR_ADMIN_KEY", // 必填，用于生成 TOKEN_KEY
      "tokenKey": "",             // 可选，自动生成并保存
      "deviceType": "ipad"        // ipad 或 mac
    }
  }
}
```

前置条件：
1. 部署 WeChatPadPro 协议服务（参考 https://github.com/WeChatPadPro/WeChatPadPro）
2. 确保服务运行在 127.0.0.1:8059
3. 获取有效的 ADMIN_KEY
4. 安装 OpenClaw（npm install -g openclaw@latest）

## 六、实施步骤

1. 复制 openclaw-wechat-main 为 openclaw-wechat-padpro
2. 实现 padpro-client.ts（封装 8059 API，KEY 管理）
3. 实现 message-listener.ts（WebSocket 连接、断线重连、消息转换）
4. 修改 config-schema.ts / types.ts / channel.ts（配置字段）
5. 修改 reply-dispatcher.ts（使用新 PadProClient）
6. 测试完整流程：登录 → 收消息 → 回复

## 七、与苍何方案的对比

| 维度 | 苍何方案 (openclaw-wechat) | 本方案 (openclaw-wechat-padpro) |
|------|--------------------------|-------------------------------|
| 费用 | 需要购买 API Key | 免费（自建协议服务） |
| 部署复杂度 | 简单（只需配置 Key） | 中等（需要部署 8059 协议服务） |
| 稳定性 | 高（代理服务维护） | 中（需要自己维护协议服务） |
| 消息接收 | Webhook 被动推送 | WebSocket 主动监听 |
| 功能完整度 | 文本+图片 | 文本+图片+语音+视频+表情+红包+朋友圈 |
| 自主可控 | 依赖第三方代理 | 完全自主 |

## 八、风险与注意事项

1. WeChatPadPro 协议稳定性 — 第三方协议可能随微信更新而失效
2. 账号安全 — 建议使用备用微信号测试，避免主号被封
3. 频率限制 — 消息发送间隔不少于 1 秒，避免触发风控
4. IP 稳定性 — 建议部署在固定 IP 的服务器上
5. ADMIN_KEY 安全 — 不要泄露，不要提交到 Git
